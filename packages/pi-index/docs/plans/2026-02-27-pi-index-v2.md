# pi-index v2: State-of-the-Art Codebase Indexing — Implementation Plan

> **For Claude:** Implement this plan task-by-task using `execute` — fresh subagent per task with three-gate review (spec → quality → security). Load the skill and begin.

**Goal:** Evolve pi-index from a solid v1 code-search extension into a state-of-the-art, production-grade indexing package for pi that handles any file type, leverages LanceDB's full capabilities, and incorporates modern retrieval techniques (contextual retrieval, parent-child chunks, scalar indexing, multi-provider embeddings, extended file support).

**Architecture:** The upgrade is organized into 5 phases, each self-contained and shippable. Phase 1 fixes the 1% spec gap and adds missing polish. Phase 2 unlocks LanceDB's advanced features (scalar indexes, BTREE filtering, table optimization, vector index tuning). Phase 3 extends file-type support beyond code to config, data, and documentation files. Phase 4 adds contextual retrieval and parent-child chunk architecture for dramatically better search quality. Phase 5 introduces multi-provider embedding support (Voyage Code-3, local models) and quantized embeddings for performance.

**Tech Stack:** TypeScript, LanceDB (^0.26+), OpenAI SDK, pi extension API. New optional deps: Voyage AI SDK (phase 5). No new required dependencies until Phase 5.

**Research Sources:** This plan is informed by deep analysis of:
- LanceDB docs (hybrid search, rerankers, scalar indexing, IVF-PQ/HNSW, FTS, table optimization)
- LanceDB vectordb-recipes (code search with tree-sitter, contextual retrieval, agentic RAG, GraphRAG tutorials)
- Anthropic's Contextual Retrieval (67% error reduction via context-augmented chunks)
- Voyage Code-3 benchmarks (13-17% improvement over generic models for code)
- Parent-child document retrieval patterns (small chunks for precision, large chunks for context)
- Matryoshka Representation Learning (variable-dimension embeddings, 67% storage reduction)
- OpenClaw's memory system (temporal decay, multi-provider, hybrid merge, query expansion)

In `db.ts`:
```typescript
/**
 * Create an IVF-PQ vector index if the table has enough rows.
 * Below the threshold, brute-force scan is faster than maintaining an index.
 * Idempotent — skips if index already exists.
 */
async createVectorIndexIfNeeded(): Promise<void> {
  await this.ensureInitialized();
  const count = await this.count();
  if (count < VECTOR_INDEX_THRESHOLD) return;

  try {
    await this.table!.createIndex("vector", {
      config: Index.ivfPq({
        numPartitions: Math.min(Math.ceil(Math.sqrt(count)), 256),
        numSubVectors: 96,
      }),
    });
  } catch (err) {
    // Index may already exist or version may not support it
    console.warn("[pi-index] vector index creation skipped:", err);
  }
}
```

In `indexer.ts`, after optimize:

 * Generate enriched text for embedding that includes file-level context.
 *
 * Format:
 * ```
 * File: src/auth/jwt.ts (typescript)
 * Module symbols: signToken, verifyToken, refreshToken
 * Imports: jsonwebtoken, ../config
 * Current: verifyToken (chunk 2 of 5)
 * ---
 * {raw chunk text}
 * ```
 *
 * This is a deterministic, zero-cost alternative to LLM-generated contextual retrieval.
 * The enriched text is used ONLY for the embedding API call — the stored `text` field
 * remains the raw source lines.
 */
export function enrichForEmbedding(chunk: CodeChunk, siblings: CodeChunk[]): string {
  const sameFile = siblings.filter(s => s.filePath === chunk.filePath);
  const symbols = sameFile.map(s => s.symbol).filter(Boolean);
  const preamble = sameFile.find(s => s.chunkIndex === 0 && !s.symbol);
  const imports = preamble ? extractImportNames(preamble.text) : [];

  const lines: string[] = [
    `File: ${chunk.filePath} (${chunk.language})`,
  ];
  if (symbols.length > 0) lines.push(`Module symbols: ${symbols.join(", ")}`);
  if (imports.length > 0) lines.push(`Imports: ${imports.join(", ")}`);
  if (chunk.symbol) lines.push(`Current: ${chunk.symbol} (chunk ${chunk.chunkIndex + 1} of ${sameFile.length})`);
  lines.push("---");
  lines.push(chunk.text);
  return lines.join("\n");
}

## Appendix: Graph Retrieval Assessment

### Should pi-index add GraphRAG or knowledge graph features?

**Recommendation: NOT in v2. Evaluate for v3.**

**Why not now:**
1. **Complexity cost is high.** Building a code dependency graph requires either AST parsing (which the spec explicitly excludes as out of scope) or an external tool like tree-sitter (new native dependency). The ROI is unclear for typical pi agent usage patterns.

2. **Contextual retrieval covers 80% of the value.** The parent-child architecture + contextual enrichment (Task 13-14) gives the agent file-level context, symbol neighbors, and import awareness — which is what graph traversal would primarily be used for.

3. **LanceDB doesn't natively support graphs.** Graph queries would require a separate data structure (adjacency list in JSON, or a dedicated graph DB like SQLite with recursive CTEs). This adds complexity without using LanceDB's strengths.

4. **The agentic pattern is already available.** The pi agent can call `codebase_search` multiple times in sequence — "find function X" → "find callers of X" → "find what module X belongs to". This is effectively agentic graph traversal without building an explicit graph.

**When to reconsider (v3 triggers):**
- Users report needing "find all callers of X" as a single tool call
- Codebases > 100K files where relationship-aware search significantly outperforms keyword + semantic
- tree-sitter becomes available as a pi peer dependency (already used by other pi extensions)
- LanceDB adds native graph query support

**Alternative for v2:** Add a `@symbol:` scope filter that does exact/fuzzy symbol matching via BM25 (no graph needed). This handles the most common "find this specific function" use case without graph infrastructure.
